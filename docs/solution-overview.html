停车明细分析报告 Agent · 实施方案

Agentic Report Generation System — 技术探讨用流程图与设计说明

栈：FastAPI · LangGraph/LangChain · PostgreSQL · pandas/matplotlib/python-docx · Docker Compose

一句话概括：上传 CSV + DOCX 模板 → 异步生成 → 轮询状态 → 下载报告。 核心设计是确定性计算与 LLM 推理的清晰切分：六个硬指标 100% 由 pandas 代码算出、LLM 不可覆盖； LangGraph 工作流带真实条件分支，Agent 据数据画像决定是否补充风险分析，再由 LLM 选择重点、组织叙述。
I/O 节点（加载/落盘） 确定性节点（代码计算事实） 条件分支（数据驱动） LLM 节点（推理/叙述）

1. 系统架构总览

三层容器（docker-compose），前端 Nginx、后端 FastAPI、PostgreSQL；后端内部再分 API → 服务编排 → Agent 工作流 → 报表流水线。

2. 异步任务生命周期与状态机

生成不阻塞提交请求；状态持久化在 DB，所以"重试容易加上"（幂等、可恢复）——这是题目要求的设计点。

queued → running → completed  ｜  failed

DB 层 CheckConstraint 约束状态只能取这四个值 · status 字段加索引

阶段谁触发写入 DB日志事件
提交POST /api/jobsinsert(status=queued, paths)job.created
开始Harness.runstatus=running, started_atjob.running
每次尝试_invoke_with_retry—job.graph_attempt.started/completed/failed · job.retry_scheduled
成功Harness.runoutput_path, status=completed, completed_atjob.completed
失败except 分支rollback→重取→status=failed, error_messagejob.failed
下载GET …/download—（只读，409 守卫未完成）job.download
演示细节 min_job_seconds：compose 里设 MIN_JOB_SECONDS=2。样例数据生成只需毫秒级，会"秒完成"导致前端看不到 running 态； Harness 在成功后补足到最小运行时长（打 job.minimum_runtime.wait），让评审能观察到 queued→running→completed 的完整流转。纯演示用，生产置 0。

3. Agent 工作流（LangGraph 状态机）

单智能体 + 有边界的状态图。profile_transactions 之后有真实条件分支，按确定性画像决定后续路径——这是"非平凡决策"而非"单次 LLM 调用"的关键体现。

load_inputs
透传初始 state（job_id / 路径 / instructions）
compute_hard_metrics
analyzer：六个硬指标 + 交易画像（确定性，LLM 不可触碰）
profile_transactions
画像就绪，作为分支判据
◇ decide_profile_branch
抵扣占比 ≥ 30%　或　存在 ≥12h 长停记录？
是 → 补充风险
enrich_risk_findings
打标记：high_deduct_ratio / long_stay_records
否 → 直接规划
plan_report_with_llm
决定章节集合；有 risk_flags 时插入"风险提示"节
draft_narrative_with_llm
生成摘要/补充观察/建议（Qwen 结构化输出，或无 key 时 stub）
generate_charts
matplotlib 出 4 张真实图（支付方式/时长/入场时段/收费时段）
render_docx
python-docx 套模板渲染，清除占位图，落盘 reports/{id}.docx
persist_success → END
返回 final_state，Harness 写 output_path 并置 completed
每个节点都由 logged_node() 包装，自动打 graph.node.started/completed/failed； 节点抛异常 → 冒泡到 Harness → job 置 failed 并记录错误。Agent loop 有边界（最多 2 次 attempt），不做无限自主循环。

4. 核心设计：确定性计算 ⟷ LLM 推理的边界

这是评分最看重的点。事实由代码算、LLM 只在 grounded context 上做表达与选择，无法改写任何硬数字。

确定性侧（代码，可复现）

六个硬指标（analyzer.compute_hard_metrics）：

指标定义
总交易笔数len(df)
应收总金额应收金额.sum()
实收总金额实收金额(元).sum()
实际抵扣总额实际抵扣额(元).sum()
实收率实收/应收×100，1 位小数
主要支付方式支付方式 众数 + 笔数

画像：数据周期、支付方式/渠道分布、停车时长(均值/最长/分箱)、逐时入场&收费分布、异常候选(长停/零应收有实收/全额抵扣)。

LLM 侧（推理，受约束）

结构化输出 ReportDraft（pydantic，extra="forbid"）：

executive_summary observations[] recommendations[]
  • Prompt 把硬指标 + 画像作为 grounded context 传入，明确"不得重算/覆盖六个硬指标"。
  • with_structured_output(ReportDraft) 强制 schema，杜绝自由文本跑偏。
  • 补充观察被引导聚焦：收入流失（应收-实收缺口≈抵扣）、优惠/免费敞口、长停异常、时段规律。
  • 无 Qwen key → stub_report_draft：同样基于真实数字生成，保证离线可评审、可测试。
关键保证：LLM 的输出只进入"叙述/观察/建议"段落；表格里的六个硬数字始终来自 analyzer，二者在 renderer 中分开写入。

6. 可观测性（结构化日志）

log_event() 统一输出单行 JSON 到 stdout（含 UTC 时间戳）；可选接 LangSmith tracing。事后可凭 job_id 串起整条链路。

类别事件关键字段
用户请求job.created · job.downloadjob_id, status
Job 生命周期job.running · job.completed · job.failed · job.missingjob_id, output_path, error
重试job.graph_attempt.* · job.retry_scheduled · job.minimum_runtime.waitattempt, next_attempt
Graph 节点graph.node.started/completed/failedjob_id, node
LLM 调用llm.call.completed / llm.call.failedmodel, prompt, response, latency_ms
题目明确要求"记录每次 LLM 调用的 prompt/response/model/latency" → 由 llm.call.* 事件满足，便于事后调试 Agent 行为。

7. 关键设计决策与权衡（讲解要点）

决策选择理由 / 权衡
异步执行FastAPI BackgroundTasks + DB 持久化不引入 Celery/Redis，控制在 6h 范围；状态持久化 → 重试/恢复"容易加"。代价：单机、进程内，不抗重启（已知，可讨论升级路径）。
Agent 框架LangGraph 单智能体状态图显式节点 + 条件边，可序列化、可调试、有边界；优于"一坨 prompt"。未上多智能体（无明确需求）。
非平凡决策profile 后条件分支抵扣占比/长停驱动是否补风险节，进而影响章节与叙述——证明 Agent 在"决定写什么"。
硬指标pandas 确定性计算 + 不硬编码同输入必同输出；LLM 仅在 grounded 事实上叙述，schema 强约束防跑偏。
LLM 可降级Qwen ↔ stub 双路无 key 也能端到端跑通、可测试、可评审；CI 不依赖外部 API。
模板策略清空重建 + 删占位图 + 复用样式章节对齐官方模板但内容全由数据驱动；模板无关性较好。
测试API 级 submit→status→download（mock harness）+ 硬指标/渲染单测满足最低要求，风险驱动 TDD 优先覆盖确定性计算与生命周期，不追求穷尽覆盖。
交付docker-compose 一键起backend 启动先 alembic upgrade head 再 uvicorn；db 健康检查后再起后端。

8. 面试预设问答（Be ready to discuss）

点开每条看要点。面试官常从"为什么这么设计 / 如何扩展 / 边界在哪"切入。

Q1. 怎么保证这是"真 Agent"而不是模板填充 + 单次 LLM 调用？
三点：① LangGraph 多节点状态图，节点间有数据流转的 state；② profile 后的真实条件分支——抵扣占比/长停决定是否走 enrich_risk_findings，进而改变 plan 的章节集合；③ LLM 承担"选重点 + 组织叙述 + 写观察/建议"两个独立节点（plan 与 draft），不是一次性产文本。硬指标完全在 LLM 之外，体现"管理 Agent 产出确定性结果"的能力。
Q2. 为什么用 BackgroundTasks 而不是 Celery？重试怎么加？
6 小时范围内不该做成分布式任务系统。关键是把状态持久化到 DB 且生成幂等（按 job_id 找输入、产出可覆盖），所以重试是"加上去"而非"重构"。
升级路径：把 run_generation_harness(job_id) 换成 Celery/RQ 任务即可——签名只吃 job_id，状态都在 DB；再加 attempts 计数列 + 指数退避 + 死信。当前 Harness 已内置 max_attempts=2 的图级重试雏形。
Q3. 如何确保六个硬指标永远正确、可复现、且 LLM 不能篡改？
analyzer 用 pandas 直接对列求和/计数，无随机性 → 同输入同输出；有专门单测锁定样例值（3674 / 105795 / 57397.5 / 48285 / 54.3% / 微信1435）。renderer 中表格数字只取 metrics 字典，LLM 的 ReportDraft 只供叙述段落，物理上分离。Prompt 也显式声明硬指标不可重算。即便 LLM 乱说，也进不了指标表。
Q4. 没有 LLM key 时怎么办？stub 会不会"假数据"？
stub_report_draft 同样读真实 metrics/profile 生成摘要、观察、建议（如"应收-实收缺口≈抵扣、占应收 45.6%"都是算出来的），只是不调外部模型。好处：离线可端到端、CI 不依赖网络、评审无需配 key。接 Qwen 时走 ChatOpenAI.with_structured_output，OpenAI 兼容接口，换 base_url/model 即可。
Q5. 补充观察的"洞察"从哪来？怎么避免空话？
把对管理者最有价值的信号预先算成事实喂给 LLM：收入流失（应收 105795 − 实收 57398 ≈ 48398 缺口，几乎全由 48285 抵扣造成，占应收 45.6%）、优惠敞口（会员积分+优惠券 2173 笔占 59.1%）、长停异常（≥12h 共 78 笔，最长 165h）。每条观察都带具体数字。Prompt 指明聚焦这些维度、必须数字支撑。
Q6. 失败如何处理？节点抛错会怎样？
节点异常被 logged_node 记 graph.node.failed 后冒泡；Harness 的 _invoke_with_retry 捕获，未达 max_attempts 则记 retry_scheduled 重试，否则抛 RuntimeError。run() 的 except 分支 rollback → 重新取 job → status=failed + error_message → commit → 记 job.failed。前端轮询到 failed 展示错误。下载端点对未完成返回 409。
Q7. 模板无关性 / 换个模板还能用吗？
当前 renderer 清空模板正文、复用其命名样式后按固定章节重建，章节标题对齐官方模板。强项：不依赖模板里的占位文字位置，稳定。局限：章节结构是代码写死的，换全新结构模板需调 renderer。可讨论的演进：让 plan 节点输出"章节计划"驱动 renderer，做到更强的模板/结构无关性——属于"可加但不在 6h 范围"。
Q8. 哪里会成为瓶颈 / 生产化还差什么？
  • 执行模型：进程内 BackgroundTasks，重启丢任务、无并发控制 → 上 Celery/RQ + 队列。
  • 存储：本地卷 → 对象存储（S3），便于多副本与下载分发。
  • 大文件：上传已分块流式落盘；analyzer 全量读入内存，超大 CSV 需分块/采样。
  • 鉴权 / 多租户：题目明确 out of scope，生产需加。
  • LLM 稳健性：超时/重试/成本上限/响应校验（已有 schema 约束）。
Q9. min_job_seconds 是什么，为什么在？
纯演示开关。样例数据生成是毫秒级，前端 1.5s 轮询可能直接看到 completed，观察不到 running 态。成功后补足到最小运行时长（默认 0，compose 设 2s），让评审看到完整状态流转，并打 job.minimum_runtime.wait 日志。生产应置 0——它不影响任何报告内容或硬指标。
Q10. 测试策略？为什么不追求高覆盖？
风险驱动 TDD：优先覆盖最易错且影响评分的部分——六个硬指标计算、API submit→status→download 生命周期（mock 掉 harness，不依赖真实 LLM）、渲染产出 docx 含图、stub 不覆盖硬指标。题目明确"不要求穷尽覆盖、不要镀金"，所以胶水代码不强测。共 11 个测试，跑通即绿。
