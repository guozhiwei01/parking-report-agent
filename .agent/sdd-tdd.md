# 轻量 SDD / TDD

使用轻量 SDD 和风险驱动的 TDD。

实现一个功能前，先明确最小规格：

- 用户可见行为是什么。
- 输入和输出是什么。
- 会改哪些持久化状态。
- 失败时怎么表现。
- 验收检查是什么。

不要为了每个小改动写大型设计文档。README 和 `.agent/` 文件就是长期规格。

这些地方优先用 TDD：

- 六个硬指标的确定性计算。
- API 生命周期：submit -> status -> download。
- job 状态流转。
- LLM 结构化输出校验。
- 报表生成失败处理。

低风险胶水代码可以先实现再补验证，但每个完整功能提交前都要有对应的验证步骤。

本作业最低测试要求：一个 API 级测试，覆盖提交、查状态、下载，并 mock 掉生成过程。
